## Rate limits

All endpoints share a single rate limit of **60 requests per minute**. Image generation uses half the rate limit (30 req/min). Video generation uses a stricter limit of **2 requests per minute** due to its higher compute cost. Polling the video status endpoint does not count toward the video rate limit.

## Rate limit headers

All responses include these headers so you can monitor your rate limit status:

| Header | Description |
|---|---|
| `X-RateLimit-Limit` | Max requests allowed in the current window |
| `X-RateLimit-Remaining` | Requests remaining in the current window |
| `X-RateLimit-Reset` | Seconds until the rate limit window resets |
| `Retry-After` | Seconds to wait before retrying (only on 429 responses) |

## Handling rate limits

When you exceed your rate limit, the API returns a `429` status code. Use the `Retry-After` header to determine when to retry:

```python
import time
import requests

response = requests.post("https://ninjachat.ai/api/v1/chat", ...)

if response.status_code == 429:
    retry_after = int(response.headers.get("Retry-After", 60))
    time.sleep(retry_after)
    # Retry the request
```

For production applications, implement exponential backoff to handle rate limits gracefully.
